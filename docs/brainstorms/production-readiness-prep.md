# SDK Production-Readiness Review: Prep & Recommendations

## 1. Purpose and status

This document is **research and recommendations for the user to refine**. It is *prep only*. It specifies the review-sweep program and the candidate hardening workstreams; it does **not** execute any sweep, does **not** produce the final plan documentation, and does **not** draft the issue-470 body. Those steps come *after* the user walks through the open questions in section 7.

Scope guardrails (fixed decisions, restated so the later phases inherit them):

- **1.0 surface** = lend + swap + borrow + wallet execution (EOA, hosted Privy/Turnkey/Dynamic, smart/ERC-4337) on Mainnet + OP Mainnet + Base. The **SDK is the core value proposition** and gets the concentrated effort.
- **Demo (`packages/demo/frontend`, `packages/demo/backend`) and CLI (`packages/cli`) are review-only plus fund-safety fixes** — no architectural refactors. Non-fund-safety findings there become backlog tickets.
- The eventual outcome of this effort is **many small, parallelizable tickets**, each pickable by a separate agent (not one agent implementing everything), and the later ticket-creation phase must **check for an existing duplicate issue before filing each ticket** (dedup-before-filing).
- Terminology: this is framed throughout as **security review / review sweep / review pass / threat model / hardening**. The word "audit" is deliberately not used.
- We hold **test-mode credentials for Privy, Turnkey, and Dynamic**, so real wallet-conformance e2e is runnable. The fork/e2e feature-test is **manual / on-demand** (not run on every deploy).

## 2. Research summary

### 2.1 Calldata integrity — recalibrated framing

Intent-integrity for the normal path is implemented **by construction**, not by a runtime decode gate:

- Concrete providers self-encode calldata from typed args via viem `encodeFunctionData` / `encodeAbiParameters` (e.g. `packages/sdk/src/actions/swap/providers/uniswap/encoding.ts`).
- The base wraps those bytes verbatim (`packages/sdk/src/actions/swap/core/SwapProvider.ts` `buildSwapTransactions` builds `{to: routerAddress, data: swapCalldata}`); `EOAWallet.send` signs `{to,value,data}` verbatim.
- Cross-namespace pre-built quotes are guarded by `WalletSwapNamespace.requireQuoteForThisWallet`, which compares the `quote.recipient` **field** to `wallet.address` and otherwise forces a re-quote.

There is **no `decodeFunctionData`-based decode-and-compare gate in production** (`decodeFunctionData` is test-only). The real, narrower gap is high-value **defense-in-depth**, not a missing core feature. But the gap has **three layers** that a naive decode-and-compare backstop does *not* all cover:

- (a) the base trusts whatever bytes a concrete `_getQuote` returns, so a buggy or hostile provider that self-encodes a wrong recipient or a near-zero min-out is not caught;
- (b) the recipient guard checks the metadata **field**, not the recipient encoded in the calldata **bytes**. Recipient location is router-type-dependent: Uniswap V4 `TAKE_ALL` sends output to `msg.sender` (recipient never in calldata); Velodrome direct v2 **bakes** recipient into bytes; Velodrome universal-router **hardcodes** the `UNIVERSAL_ROUTER_MSG_SENDER` sentinel and discards the caller recipient (confirmed `velodrome/encoding/routers/cl.ts:152`, issue 444);
- (c) **the load-bearing design subtlety**: several intent fields are *resolved or derived before encoding*, so bytes equal the already-corrupted intent and a decode-and-compare against the resolved intent passes. The min-out is derived from the same quote that drives display (and the Uniswap path *re-derives* it a second time inline — see 2.5); the recipient may be **ENS-resolved** through a mainnet RPC (2.6); the actions-namespace recipient may be **defaulted to a router sentinel** (`SwapProvider.ts:274`, see 2.6). A decode-and-compare gate is only as strong as the intent it compares against: if it compares against the post-resolution/derivation intent, it cannot catch resolution-layer or derivation-layer tampering. Whether the backstop compares against raw user input or resolved intent is the single most important design question for the Tier-1 centerpiece (question 1, 7.1).

### 2.2 The shared structural blind spot

Both **code review** and **honest-Anvil e2e** only catch what is visibly wrong or fails to execute. **Self-consistent failures pass both.** The canonical instance: min-out is derived from the same quote that drives the display, so a wrong-but-self-consistent floor (or a lying RPC inflating the quoter output) settles below market while passing every check. Eight residual fund-loss vectors share this shape, each broken only by a technique that introduces independence: differential/independent decode, adversarial-RPC fault injection, price-perturbation fork scenarios, malicious-bundler conformance, quote-aging tests, an adversarial-token suite, supply-chain independence, and a decode-bytes recipient assertion. The blind spot also reappears *at the assertion layer*: an exact-amount fork assertion with too loose a tolerance re-admits a self-consistent skim (a 1% band hides a 0.9% skim), so the tolerance band is itself a fund-safety decision (question 22).

### 2.3 The two unguarded recipient-redirection surfaces (newly surfaced)

Two shipped surfaces move or redirect funds **outside** the swap/lend/borrow quote-and-encode pipeline, so none of the calldata-centric mitigations reach them:

- **Bare wallet transfer / send / sendBatch.** `EOAWallet.send` (`eoa/EOAWallet.ts:62`) and `sendBatch` (`:90`) sign `{to, value, data}` verbatim with **no quote, no market, no recipient guard, no allowlist**. A plain native or token transfer is the *cheapest* fund-loss surface and the *least* guarded: the bytes ARE the intent, so a decode-and-compare backstop structurally cannot catch a wrong-recipient transfer (a typo, an address-poisoning swap, or a compromised caller). S8 frames this path as reliability/verbatim-signing; this prep adds an explicit fund-safety framing and an open question (question 4 in 7.1) on whether the SDK owes the transfer primitive any recipient-confirmation / address-poisoning defense at all, or whether recipient correctness is wholly the integrator's responsibility.
- **ENS / name resolution.** Swap params accept `recipient?: Address | EnsName` (confirmed `types/swap/base.ts:95,170`). `WalletSwapNamespace.resolveRawParams` (`:108-114`) and the base namespaces call `resolveRecipient` → `resolveAddress` (`services/nameservices/ens/utils.ts:43`), which resolves a human-readable name to an address **via a mainnet `PublicClient`** and bakes that address into calldata as the swap output recipient. This is a classic fund-redirection vector (poisoned/wildcard CCIP-read resolver, normalization mismatch, cache poisoning, compromised mainnet RPC). It happens **before** encoding, so bytes == metadata == attacker address and the decode-and-compare backstop passes. The `requireQuoteForThisWallet` / `isAddressEqual` guard (`WalletSwapNamespace.ts:96`) fires only on the pre-built-`SwapQuote` branch, **not** the typed-`WalletSwapParams` branch that resolves the ENS recipient and uses it directly. This is the off-chain-resolution analogue of the Morpho-GraphQL trust already covered for lend — but for the single field that controls where funds land. It needs its own threat-model invariant and a clear sweep owner (S3, with the resolver path pulled out of S11's hygiene scope).

### 2.4 The actions-namespace sentinel-recipient default

`SwapProvider.resolveParams` substitutes a router sentinel when no recipient is supplied: `const recipient = params.recipient ?? UNIVERSAL_ROUTER_MSG_SENDER` (`SwapProvider.ts:274`). On the **actions** (non-wallet) path there is no wallet to bind to, and the `isAddressEqual` guard lives only in `WalletSwapNamespace`. So an actions-namespace caller who omits recipient gets sentinel-routed output with no binding to any user address and no guard. Issue 435 (sentinel leak) is the metadata/`getQuote` face of this; the deeper question is whether the actions path should **fail closed** when recipient is unset rather than silently defaulting to a router sentinel (question 6 in 7.1).

### 2.5 Uniswap min-out double-derivation

The min-out is computed **twice, independently**, from the same inputs. `UniswapSwapProvider._getQuote` calls the base `SwapProvider.computeSlippageBounds` (`SwapProvider.ts:286-297`), while `encodeUniversalRouterSwap` **re-derives** it inline with a float-first formula: `(quote.amountOutRaw * BigInt(Math.round((1 - slippage) * 10000))) / 10000n` (`uniswap/encoding.ts:251-252`). For normal inputs the two agree, but **there is no assertion that they agree**, and the inline path carries float-rounding edge-case risk. The distinct S4 correctness target is this double-derivation consistency assertion plus the precision risk in the inline formula — *not* a second independent review of `computeSlippageBounds` (which is base-class code owned by S1).

### 2.6 The shared structural blind spot — assertion-layer corollary

(Folded into 2.2 above; retained as a pointer.) Exact-amount fork assertions must recompute expected output **independently of `quote.amountOut`** (from on-chain reserves/oracle) and use a tolerance band tight enough to detect a value-skim — otherwise the self-consistency blind spot simply moves into the test harness.

### 2.7 PR 348 — adopt-with-changes

PR 348 (open, targets issue 335, `feat/e2e-system-tests`) is the right chassis: a shared `packages/sdk/src/test/network/` harness, `TestEOAWallet` subclassing the real `EOAWallet`, a real `ChainManager` on the fork, an any-free `ForkClient`, pinned fixtures, CI fail-fast `getRpcUrl`, and `expectReceiptSuccess`. It ratifies the **capability-boundary-once** philosophy (not the full wallet × chain × function product). **Verified on this branch: `packages/sdk/src/test/network/` does not yet exist**, so landing PR 348 is the **blocking first step** for every execution-bearing sweep. Required fixes before it is a confidence signal: (i) dead-on-arrival lend tests (Aave built with empty `{}`, no `marketAllowlist`, throws `ProviderNotConfiguredError` before executing); (ii) a string-coercion no-op assertion (`toBeGreaterThan(0)` on a `*Formatted` string); (iii) fixed ports 18544-46 collide under vitest file parallelism; (iv) directional-liveness rather than exact-amount/adversarial assertions; (v) two divergent fork harnesses remain (`MorphoBorrowProvider.network.test.ts`, `VelodromeSwapProvider.network.test.ts` — both confirmed present and using the `as unknown as ChainManager` fake); (vi) namespace execution is OP-only; (vii) whale-funding is USDC-only and fragile.

### 2.8 Existing-controls state

- **Recipient-binding guard**: present, but **metadata-level only**, and only on the pre-built-quote wallet branch — **not** the typed-params branch that resolves ENS, and **not** the actions-namespace path that defaults to a sentinel.
- **Morpho marketId keccak verify**: present (borrow only); **no Aave/lend/swap equivalent** — Aave borrow uses a *synthetic* self-computed marketId.
- **Market allowlist / blocklist**: borrow enforces fail-closed (`borrow/core/validations.ts`, confirmed: empty allowlist fails closed, blocklist enforced). **Lend reads only `marketAllowlist`, never `marketBlocklist`** — the `marketBlocklist` field exists in the type (`types/lend/base.ts:201`) but is never read in `lend/core/LendProvider.ts`, and an empty allowlist allows all markets (issue 334, an asymmetric control gap). Swap `validateMarketAllowed` **does** read `marketBlocklist`. The **asset-level** `assets.block` field is *not* inert — `getSupportedAssets()` filters it (`actions.ts:198`); the inert construct is specifically the lend `marketBlocklist`. (This precision matters: an S6 subagent must hunt the unenforced lend `marketBlocklist`, not a non-existent missing asset-level blocklist.)
- **Boundary validation + approve utils**: present (syntactic/checksum only); approve defaults exact; Permit2 30-day expiry. Permit2 is **on the signing path** (a SignatureTransfer/AllowanceTransfer signature is a signed off-chain pull-authorization), not only the approval path — so witness/nonce/deadline are decode-before-sign targets too (folded into Tier 1).
- **Named errors**: partial and **smaller than four modules tells you**. There are at least four named-error modules — `core/error/errors.ts`, `wallet/core/error/errors.ts`, `wallet/core/wallets/smart/error/errors.ts`, `services/nameservices/ens/errors.ts` (confirmed) — plus the borrow domain's `validations.ts` error usage. But **swap and lend have NO named-error modules at all** (no `error/` directory under `actions/swap` or `actions/lend`, confirmed); they throw bare `Error`. `DefaultSmartWallet.send` additionally throws a bare `Error` concatenating `err.message`, dropping both cause and class — the worst discipline on the most critical path. The taxonomy workstream is therefore "**add named-error modules to swap + lend** (the SDK side of issue 474), reconcile with the core/wallet/smart/ens modules, and fix the `DefaultSmartWallet.send` collapse" — not "fix three modules."
- **CI** (`.circleci/config.yml`): partial. No network/fork job (`pnpm test:network` runs in zero jobs), no dependency or secret scan in the PR pipeline, no `--max-warnings 0` gate (so issues 337/381 can grow), node-version skew (checks on 20.11, build on 22.14). Wiz (issue 432) is server-side, supply-chain/secrets only, not PR-blocking. `fast-check` is absent (confirmed). The public-export surface test is borrow-only (issue 483). The issue-131 build-without-all-deps test is missing.

## 3. Recommended workstreams

Workstreams map to the tiered approaches menu. Each lists a few high-level candidate tickets with fund-safety leverage noted. **Detailed ticketization is the later phase** — the property lists, exact seams, and per-function targets below are *direction*, not commitments; the sweeps should confirm where the gaps actually are before the plan doc pins them.

### Tier 1 — highest leverage (independence-breaking)

- **Decode-and-compare calldata backstop.** Add a base-level decode of provider-self-encoded calldata to verify recipient-in-bytes, min-out, and token before the wallet wraps it verbatim — at **both** wallet seams (`EOAWallet.send` and `DefaultSmartWallet.send`/`sendBatch`), not only `SwapProvider`. **Scope explicitly includes Permit2 signature payloads** (witness/nonce/deadline/spender), not only transaction calldata, since a Permit2 signature is a signed pull-authorization on the signing path. **Design gate (must be answered first):** does the comparison run against *raw user input* (pre-resolution) or *resolved/derived intent*? If the latter, it cannot catch ENS-resolution, sentinel-defaulting, or min-out-re-derivation tampering, and a separate independent-derivation/independent-resolution check is required. Candidate tickets: base `SwapProvider` seam backstop (extends issue 373); wallet-dispatch decode-before-sign (sibling of 373, covers lend/borrow too); per-router recipient-in-bytes assertion; Permit2-signature decode-and-compare. *Fund-safety leverage: high.*
- **Property/invariant fuzzing (`fast-check`).** None exists today. Targets (direction, to be confirmed by S4/S6): `utils/assets.ts` `parseAssetAmount`, `utils/validation.ts` `validateSlippage`, `SwapProvider.computeSlippageBounds`, the BPS math (including the Uniswap inline re-derivation in `uniswap/encoding.ts:251`), `utils/erc4626.ts`, and the `number→bigint` precision hazard behind issue 379. *High.*
- **Differential testing of our calldata vs protocol-SDK encoders.** `@aave/*` and `@morpho-org/*` are already in `node_modules`; Uniswap SDK too. Candidate tickets: sqrt/price-impact parity vs Uniswap SDK (extends issue 318); decode-and-compare our Aave/Morpho calldata vs the protocol encoders. *High.*

### Tier 2 — adversarial inputs

- **Adversarial-RPC fault-injection harness** at the `ChainManager.getTransportForChain` seam (a viem transport that can lie/delay/reorder for chosen `eth_call` selectors), driven against `uniswap` getQuote, aave `fetchAaveStateAndPrices`, **`services/tokenBalance.ts`** (balances feed max/all-in flows: "swap all", "repay max"), **and the ENS mainnet resolver** (a lying resolver redirects the recipient). *High.*
- **Malicious-bundler / decode-before-sign conformance** for ERC-4337 (`DefaultSmartWallet`). *High.*
- **Adversarial-token suite** (fee-on-transfer / rebasing / zero / dust / over-decimal). *High.*
- **Quote-aging + price-perturbation fork scenarios** (`validateQuoteNotExpired` uses wall-clock `Date.now()`, price never re-checked). *High.*

### Tier 3 — e2e + signing threat model

- **Capability-boundary e2e matrix on PR 348** with **exact-amount** assertions (recompute expected output independently of `quote.amountOut`, from on-chain reserves/oracle), extended to Mainnet + OP + Base with per-chain whale/asset fixtures. *High.*
- **Hosted + smart wallet conformance** using the in-hand creds (Dynamic is React-only — needs a react/jsdom or browser lane). **This leg contains hidden test-infrastructure** (a local bundler + paymaster sandbox for 4337, a Dynamic react/jsdom lane, archival per-chain RPC contexts) that, like the S12 fork harness, is a prerequisite *build*, not a review pass — see the timebox/DAG note in section 4. *High.*
- **Signing-path threat model**: replay rejection, wrong-owner rejection, chainId + EntryPoint binding (and **EIP-155 chain-binding on the non-4337 EOA path** — sign-for-chain-A / broadcast-on-chain-B is a fund-loss vector for any wallet type), attribution-suffix integrity on a fork (the 16-byte suffix is concat'd via `concatHex` after `prepareUserOperation` and the `userOpHash` is computed over the modified callData; verify against the smart-wallet contract's hash path and that a malformed suffix slipping past `isValidAttributionSuffix` cannot cause a silent revert), Permit2-signature integrity. *High.*

### Tier 4 — defense-in-depth + gates (hardening of existing controls only)

This tier is **hardening of code that already exists**. New enforcement primitives and shippable features are deliberately *not* here — they are in the "candidate follow-on features" list below.

- **Supply-chain hardening**: `pnpm audit` + lockfile integrity + Socket/provenance in CI; demote heavy protocol SDKs (`@aave/*`, `@morpho-org/*`) to **dev-only differential oracles** (in/out of scope is question 25); reconcile with Wiz (issue 432); add the issue-131 build-without-all-deps test.
- **Static-analysis CI gates**: `--max-warnings 0`, `ts-prune`, type-coverage, custom static-analysis rules on the calldata/signing seam, a never-mock-encoding lint rule.
- **Fail-closed defaults + tightened error taxonomy**: **add named-error modules to swap + lend** (SDK side of issue 474; S11 owns cross-module consistency, S8/S10 surface path findings only — see section 4 handoff note); fix the bare `Error` on `DefaultSmartWallet.send`; **enforce the existing-but-inert lend `marketBlocklist`** and make the lend empty-allowlist fail closed (this is *fixing a partial control*, not adding a new one).
- **Public-API + safety-defaults snapshot tests** (extend `src/__tests__/index.exports.spec.ts` beyond borrow, issue 483).
- **`THREAT_MODEL` / `INVARIANTS` doc** (executor-vs-recipient invariant, recipient-in-bytes truth table, advisory-HF limitation, 4337 binding, ENS-resolution trust assumption, single-transport trust assumption). This is a text artifact with no external commitments — legitimately in scope.

**Candidate follow-on features (OUT of the review-sweep scope; substantiated only if sweep findings justify them):** pre-sign transaction simulation/preview seam; runtime guardrails (spending caps, recipient pinning, runtime deny-list activation); dispatch-chokepoint observability pipeline; the additive `*Raw` bigint API (issue 379). These are net-new shippable subsystems requiring API surface, versioning, and consumer-migration decisions that exceed the ~1-week timebox and the "review + fund-safety fix" framing. They are listed here so the execution phase does not mistake them for hardening tickets. **`SECURITY.md` / responsible-disclosure policy** is org/process work (legal/comms/process commitments) — file it as a single small ticket assigned to a human owner, not an agent-deliverable sweep artifact.

## 4. The review-sweep program (centerpiece)

**Program shape.** 15 candidate sweeps: **11 SDK (S1–S11) + 1 network-harness foundation (S12, contingent) + backend (S13) + frontend (S14) + CLI (S15)**. The harness-foundation "sweep" is included because PR 348's `src/test/network/` does not yet exist and five execution-bearing SDK sweeps depend on it — but it is a build/infra milestone whose artifacts are a design spec + fix list + assertion contract, *not* vulnerability findings, so whether it counts as a sweep or a prerequisite milestone is question 12. If reclassified as a milestone, the SDK sweep count is **11 (S1–S11)**, still satisfying the ≥10 constraint. The `sweepCount` field below reports **15** as the upper bound with S12 counted; treat it as **14 + 1-contingent**.

**Two structural facts the flat table must not hide:**

1. **This is a two-wave DAG, not 15 co-equal parallel sweeps.** Landing PR 348 + S12's assertion contract is the blocking first step for every *execution-bearing* leg (S1/S4/S5/S6/S7/S8/S9/S10 plug into the exact-amount/adversarial contract). The honest structuring: split each execution-bearing sweep into a **static-review lane** (code reading, decode-gap analysis, threat-model authoring — independent, runnable in wave 0 *alongside* S12) and an **execution lane** (live-fork assertions — gated on S12 + PR 348 in wave 1). The static review of all surfaces is plausibly one week of parallel agents; the execution/conformance legs are **not**, because S9/S10/S12 each contain multi-day infra builds (bundler+paymaster sandbox, Dynamic react/jsdom lane, harness migration of two divergent fork harnesses). Week 1 realistically delivers **static findings + the harness**; execution lands after. This static-vs-execution split is question 13.

2. **Lens-executability caveat (must be resolved before any sweep runs).** The available-skills list exposes `compound-engineering:ce-code-review`, `ce-doc-review`, `ce-debug`, `ce-plan`, etc. The `ce-*-reviewer` persona names used in the table below (`ce-security-reviewer`, `ce-correctness-reviewer`, `ce-adversarial-reviewer`, `ce-data-integrity-guardian`, `ce-reliability-reviewer`, `ce-api-contract-reviewer`) are **internal personas that `ce-code-review` spawns, not directly-invocable skills.** A sweep agent cannot invoke them as a "lens." Either (a) re-map each CE-persona cell to "run `ce-code-review`, steering its persona set toward X," or (b) replace them with the provably-distinct granular skills (`engineering-skills:code-reviewer`, `engineering-skills:adversarial-reviewer`, `engineering-skills:senior-security`). This affects 11 of 15 sweeps. The engineering-skills lenses (`red-team`, `senior-backend`, `senior-security`, `senior-frontend`, `senior-qa`, `tdd-guide`, `security-pen-testing`) and ethskills (`security`, `wallets`, `frontend-ux`) **do** map to real skills. Note also: **"ethskills(DeFi)" is a family label, not a skill** — the invocable skills are `ethskills:security` / `wallets` / `frontend-ux`. This is question 12 (consolidated program-shape decision).

**Lens-diversity honesty.** Lenses are mixed across families *in intent*. But once CE personas collapse onto `ce-code-review`, the *effective* family diversity is lower than the table implies, and the crown-jewel triangulation claim should be re-anchored on provably-distinct skills. **S1 as drafted pairs two same-family security reviewers** (`engineering-skills:red-team` + `ce-security-reviewer`); for the single highest-value seam it should instead triangulate a genuinely orthogonal set — an adversarial-security lens + a protocol-correctness/DeFi-semantics lens (`ethskills:security`) + the differential-encoding technique. This is reflected below and surfaced as question 14.

## 5. Fund-safety threat model (concise)

**Assets at risk:** user ERC-20/native balances during swap/lend/borrow/repay **and bare transfer**; collateral health in borrow positions; allowances granted to routers/pools/Permit2.

**Trust boundaries:** (1) the concrete provider's self-encoded bytes (base trusts them verbatim); (2) the RPC/quoter/oracle — a single viem fallback transport, failover-only, never cross-checked — **which also serves the ENS mainnet resolver and `tokenBalance.ts`**; (3) the ERC-4337 bundler — `prepareUserOperation` output trusted verbatim; (4) the hosted signer (Privy/Turnkey/Dynamic) and the caller-supplied address bound to it; (5) off-chain market/APY/rewards APIs (Morpho GraphQL); (6) the **ENS resolver** (resolves a name to the recipient address before encode); (7) demo-backend request params and the faucet signer.

| Invariant | Enforced today? |
|-----------|-----------------|
| Signed calldata bytes match the typed intent (recipient/amount/min-out/token) | **No** — by-construction only; no decode-and-compare gate |
| Decode-and-compare (if added) compares against *raw user input*, not already-resolved intent | **N/A today** — design-open; if it compares resolved intent it misses ENS/sentinel/min-out-re-derivation tampering |
| Recipient in calldata **bytes** equals the intended address | **Partial** — metadata field checked on pre-built-quote branch only; bytes not (router-dependent; universal-router discards recipient, 444); ENS-resolved + actions-sentinel branches unchecked |
| ENS-resolved recipient round-trips through an independent resolver or is pinned to executor | **No** — single mainnet RPC, no cross-check, resolution precedes encode |
| Bare transfer/send recipient is verified (typo / address-poisoning) | **No** — verbatim `{to,value,data}`, no guard, decode-and-compare cannot apply |
| Swap settles at/above an independent min-out floor | **No** — min-out and display derive from the same quote (and Uniswap re-derives it inline); no oracle floor or impact gate |
| Tx value/gas parameters match intent (no native-value or gas override drift) | **No** — `TransactionOptions` declared but never applied (394); 4337 undershoot reverts after paymaster engaged |
| Signed tx binds the intended chain (EIP-155 / userOp chainId) | **Partial/Unverified** — 4337 binding only mock-tested; EOA EIP-155 chain-binding unreviewed (sign-A/broadcast-B risk) |
| Borrow projected HF cannot fall below the safety buffer | **No** — advisory off-chain projection; no on-chain min-HF arg; no path refuses unsafe calldata |
| Deny-listed / non-allowlisted markets are unreachable | **Borrow: yes (fail-closed). Lend: no** (empty-allowlist allows all; `marketBlocklist` field never read). Asset `assets.block` IS enforced |
| marketId is verified against on-chain identity | **Morpho borrow: yes. Aave: synthetic. Lend: no** |
| userOp binds chainId + EntryPoint, non-replayable, wrong-owner rejected; attribution suffix integral | **Unverified** — only mock-based wiring tests exist |
| Permit2 signature payload (witness/nonce/deadline/spender) matches intent before signing | **No** — on the signing path, not covered by any decode gate |
| Approvals sized to exact need; no orphaned allowance after partial batch | **Mostly exact**; Permit2 30-day expiry; fee-on-transfer breaks sizing; partial-batch failure orphans an exact allowance (no EOA atomicity) |
| Errors on the signing path preserve class + cause (fail-loud) | **No** — `DefaultSmartWallet.send` collapses to a bare string; swap/lend have no named errors at all |

## 7. Open questions for the user

Consolidated and de-duplicated across all scouts and critiques. Grouped by theme; each is standalone and answerable in one decision.

### 7.1 Fund-safety policy (what the SDK enforces vs surfaces)

1. **Decode-and-compare scope and reference point.** Should the backstop (a) be a hard gate that *refuses to sign* on mismatch, or observability-only; and (b) compare against *raw user input* (pre-resolution) or against the *resolved/derived intent*? If it compares resolved intent, it cannot catch ENS-resolution, sentinel-defaulting, or min-out-re-derivation tampering, and a separate independent-resolution/independent-derivation check is needed. Does its scope include **Permit2 signature payloads**, not only transaction calldata?
2. **Borrow health-factor:** ship a fail-closed guard that refuses to emit borrow calldata projecting HF below the safety buffer, or keep it advisory? If fail-closed, do we add a second price source?
3. **Lend empty-allowlist-allows-all + unenforced `marketBlocklist` (334):** fix in the 1.0 surface (fail closed, enforce the `marketBlocklist` field) or document only? (Note: the asset-level `assets.block` is already enforced; this is specifically the lend `marketBlocklist`.)
4. **Bare transfer / send recipient:** should the `transfer`/`send` primitive get any SDK-level recipient-confirmation or address-poisoning defense, or is recipient correctness entirely the integrator's responsibility? (Decode-and-compare cannot help here — the bytes are the intent.)
5. **Recipient model:** pin recipient to the signer (executor == recipient), or keep it configurable with a bytes-level assertion? Forward-account for session keys (403) where executor != recipient.
6. **Actions-namespace sentinel default:** should the actions `getQuote`/`buildSwap` path **fail closed** when recipient is unset, rather than silently defaulting to `UNIVERSAL_ROUTER_MSG_SENDER` (`SwapProvider.ts:274`)?
7. **Velodrome universal-router recipient discard (444):** is "always deliver to msg.sender, distinct recipient unsupported for that router type" intended product behavior (assert-and-document), or a latent bug (assert-and-fix)?
8. **Fee-on-transfer / non-standard tokens:** in scope for 1.0 (allowlist/handling), or explicitly out (fail-loud and document)?

### 7.2 Trust posture (RPC, bundler, signer, name resolution)

9. **RPC trust as a 1.0 question (whether before how):** for 1.0, do we **document the single-transport trust assumption** (quoter/oracle/balance/ENS all over one un-cross-checked viem fallback transport) as a known limitation and defer hardening, or is RPC-trust hardening a 1.0 *blocker*? 10. **ENS-resolution trust boundary:** is ENS recipient resolution a fund-safety trust boundary requiring a dedicated sweep lane and a pinned/independent-resolver invariant, or is the mainnet resolver RPC trusted infrastructure like the quoter RPC?

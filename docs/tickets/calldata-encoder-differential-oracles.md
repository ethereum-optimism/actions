## Problem

Every test that "validates" the calldata a user signs on the swap and lend surfaces is encoder-vs-itself: it compares the hand-rolled encoder's output to the hand-rolled encoder's own input (or decodes with the same ABI/module that did the encoding). The misleadingly-named `uniswap/__tests__/sdk.test.ts` imports only `viem` and the SDK's own `encodeUniversalRouterSwap`/`getQuote`/`calculatePriceImpact` — there is no `@uniswap/*` import anywhere in the test tree to serve as an external reference. The result is a self-consistent-encoder blind spot: the encoder is proven internally consistent, never proven correct against the canonical protocol encoding the on-chain contract expects.

The fund-safety consequence is that a whole cluster of already-ledgered fund-loss / malicious-sign mechanics is **structurally un-falsifiable by the current suite**. The swap encoder's recipient routing (V4 `TAKE_ALL` `0x0f` sends output to `msg.sender`; the universal-router and Velodrome CL/universal encoders hard-code the `UNIVERSAL_ROUTER_MSG_SENDER` sentinel and ignore the caller's `recipient`) is never decoded back out, so a swap where the caller passed `recipient = B` but funds route to `msg.sender = A` passes green. On the lend side, Morpho deposit/withdraw calldata is taken straight from the third-party `MetaMorphoAction.deposit/withdraw`, mocked in tests to a literal hex string, so the `receiver`/`owner`/`amount` the user signs is never asserted; the Aave provider hand-encodes `supply`/`withdraw`/`depositETH`/`withdrawETH` against a local `POOL_ABI`/`WETH_GATEWAY_ABI` and no test ever decodes the produced bytes to confirm `onBehalfOf`/`to`/`pool`/`amount` (the native-ETH `to` is the highest-blast-radius field with zero coverage).

The fix is to break the self-consistency: for the same intent, encode the calldata **both** via our encoder **and** via the canonical protocol reference (the Uniswap SDK as a dev-only oracle for swap; the `@aave/*` and `@morpho-org/*` SDKs already in the dependency closure for lend), then assert byte/field equality on the security-critical fields (recipient, min-out / max-in, amounts, `marketId`, `onBehalf`/`receiver`/`owner`, `deadline`). This replaces the encoder-asserts-itself tests with an independent oracle. The borrow surface already does this — `borrow/providers/morpho/__tests__/MorphoBorrowProvider.spec.ts` decodes its produced bundle with `blueAbi` imported from `@morpho-org/blue-sdk-viem` — so this is a sibling-consistency gap, not a new technique.

This ticket is a **test-infra** ticket. It does not change the production encoders; it gives the signing-path encoders the external correctness anchor they lack and produces the interim, no-bundler decode oracles that the consolidated `e2e-anvil-feature-test` depends on (that ticket lists this one as a blocker and describes its role as the independent-ABI decode of the produced swap / `executeBatch` / borrow-bundle calldata). The fund-loss *logic* fixes themselves (F046/F003/F047 recipient routing, etc.) are owned by their own tickets; this ticket makes those fixes verifiable and makes their absence fail CI.

## Findings

- **F153** (medium, correctness) — `packages/sdk/src/actions/swap/providers/uniswap/encoding.ts:1-30,224-294`; `providers/velodrome/encoding/routers/v2.ts`; `providers/velodrome/encoding/routers/cl.ts`: all Uniswap/Velodrome router calldata is hand-rolled on viem primitives with no official-SDK differential oracle; `uniswap/__tests__/sdk.test.ts` validates the encoder only against itself, leaving the F046/F047/F004/F005/F048 encoding cluster with no external reference anchor.
- **F180** (high, fund-loss) — `packages/sdk/src/actions/swap/providers/uniswap/__tests__/sdk.test.ts:261-431`: the `encodeUniversalRouterSwap` (V4) tests assert only the calldata prefix/length/inequality and the action-code bytes (`0x060c0f` / `0x080c0f`), never decode the `SETTLE_ALL`/`TAKE_ALL` recipient params; all 8 tests pass `recipient: '0xrecipient'` but none decode it back out, so the F046 recipient-ignored fund-loss bug is invisible.
- **F181** (low, fund-loss) — `packages/sdk/src/actions/swap/providers/velodrome/__tests__/encoding.v2.test.ts:177-231` (and `encoding.cl.test.ts:21-98`): the universal-router encoder test asserts only `commands==0x08`, inputs length, deadline, `payerIsUser==true` and never decodes `V2_SWAP_EXACT_IN_INPUT_PARAMS[0]` (recipient), while the sibling v2/leaf tests assert `args[3] === RECIPIENT`; the asymmetric omission is exactly the path where the F003/#444 recipient-sentinel bug ships green.
- **F189** (medium, correctness) — `packages/sdk/src/actions/lend/providers/morpho/__tests__/MorphoLendProvider.test.ts:16-23,186-213,86-114`: the Morpho deposit/withdraw signing path has zero calldata-arg verification — `MetaMorphoAction` is mocked so `deposit` returns `'0x1234567890abcdef'` and `withdraw` returns `'0xabcdef1234567890'`, so the `receiver`/`owner`/`amount` in the signed tx are never asserted; a caret-dep regression that swapped `withdraw`'s `receiver`/`owner` would pass every lend test.
- **F190** (high, malicious-sign) — `packages/sdk/src/actions/lend/providers/aave/__tests__/AaveLendProvider.test.ts:125-205,213-271,355-375`: Aave `supply`/`withdraw`/`depositETH`/`withdrawETH` calldata is never decoded in any test; the `onBehalfOf`/`to`/`pool`/`amount` args (incl. the native-ETH `to` recipient-in-bytes) in the signed tx are unverified — `position.data` is never passed to `decodeFunctionData` anywhere in the suite.
- **F160** (low, correctness) — `packages/sdk/src/actions/lend/providers/morpho/MorphoLendProvider.ts:1,64,110`: Morpho signing-path calldata is taken straight from `MetaMorphoAction.deposit/withdraw` with no `decodeFunctionData` round-trip asserting the encoded args match the caller-supplied `amount`/`receiver`/`owner`; asymmetric with the Aave sibling's local pinned ABI and with the borrow-Morpho sibling that already decodes with `blueAbi`.

## Root cause

There is no independent oracle on the swap and lend encoding paths, for two distinct reasons:

- **Swap (F153/F180/F181):** the router encoders are deliberately hand-rolled on viem with vendored ABI fragments and no `@uniswap/*` runtime dependency. This is a sound supply-chain reduction, but it means the only thing that could prove the hand-rolled bytes match Uniswap's reference encoding — the canonical SDK — is absent from the test tree. The unit tests then stop one decode-level short: they decode the action-code bytes (which prove *which* actions run) but never the action *params* (which prove *where output goes* and *how much is protected*). Asserting the encoder's output against the encoder's own input is a tautology; it can never catch a recipient sentinel, a wrong min-out, or a wrapped amount.

- **Lend (F189/F190/F160):** Morpho delegates the signed bytes entirely to the third-party `MetaMorphoAction`, mocked to a constant in tests, with no decode-back; Aave hand-encodes against a local ABI but no test decodes the produced bytes at all. Both sides of any existing round-trip use the same ABI/module that did the encoding, so a wrong param order, type width, swapped `receiver`/`owner`, or drifted local ABI round-trips cleanly. The borrow surface already established the correct pattern (decode the produced bundle with `blueAbi` from `@morpho-org/blue-sdk-viem` as the independent oracle); lend simply never adopted it.

The common thread: a green CI on these surfaces currently means "the encoder is self-consistent," not "the signed calldata routes the right amount to the right recipient against the contract the chain runs."

## Recommended approach

Test-infra only. No production encoder change is in scope here — this ticket adds differential and decode-based tests that anchor the existing encoders against an external reference, in keeping with the scope rules (this is differential testing / hardening that fails closed where the SDK already knows the intended field values; it is not intent-guessing, not a broad refuse-to-sign, and does not trust the RPC).

1. **Swap — Uniswap differential oracle (F153, F180).** Add a **dev-only** differential test (the reference SDK goes in `devDependencies`, never the runtime closure — preserve the lean runtime). For representative intents (exact-in, exact-out, native-in, and an explicit non-`msg.sender` recipient), encode the Universal Router / V4 calldata with the canonical Uniswap SDK (`@uniswap/v4-sdk` / `@uniswap/universal-router-sdk` / `@uniswap/sdk-core`) and assert byte-equality against our `encodeUniversalRouterSwap` output. Where the reference SDK cannot be made to produce byte-identical calldata (e.g. our encoder cannot encode a non-`msg.sender` recipient, which is the live F046/#444 bug), assert at the **field** level: decode the `SETTLE_ALL`/`TAKE_ALL` `CURRENCY_AMOUNT` params out of our calldata and assert the take recipient, the `amountOutMinimum` (exact-in) / `amountInMaximum` (exact-out), and the amounts — and assert our encoder **throws** when `recipient != msg.sender`, turning #444 into a failing test rather than a silent pass. Note: `@uniswap/*` is **not** currently in `pnpm-lock.yaml`, so adding the oracle means adding a new devDependency; pin it and keep it out of `dependencies`.

2. **Swap — Velodrome recipient-field symmetry (F181).** Add the missing recipient-field decode to the universal-router and CL encoder tests so they match the v2/leaf siblings that already assert `args[3] === RECIPIENT`. Decode `V2_SWAP_EXACT_IN_INPUT_PARAMS[0]` (and the CL equivalent) and assert the encoded recipient. Given the sentinel design, assert the value equals `UNIVERSAL_ROUTER_MSG_SENDER` **and** add a case asserting a non-sentinel caller recipient is rejected/surfaced, so the F003/#444 silent-drop becomes a failing assertion. Where a Velodrome/Aerodrome reference encoder exists, mirror the Uniswap differential approach; otherwise the symmetric field-decode is the oracle.

3. **Lend — Morpho decode-back with `blue-sdk` ABI (F189, F160).** Replace the `MetaMorphoAction`-mocked literal-hex tests with a test that does **not** mock the module: call `_openPosition`/`_closePosition`, then `decodeFunctionData` the produced bytes against the independent `erc4626Abi` / `blueAbi` (`@morpho-org/blue-sdk-viem`, already in the closure and already used this way by the borrow sibling) and assert `functionName`, `args.assets === amountWei`, `args.receiver === wallet`, `args.owner === wallet`, and `transaction.to === vaultAddress`. The reference ABI is the oracle, not the same `MetaMorphoAction` that encoded it.

4. **Lend — Aave decode-back against the protocol selectors (F190).** Add decode-based assertions for each of `supply`/`withdraw`/`depositETH`/`withdrawETH`: decode `position.data` and assert `functionName` plus **every** arg — `asset`, `amount`, `onBehalfOf`/`to === wallet`, `pool === poolAddress`, `referralCode === 0` — with the native-ETH `to` (who receives native ETH on `withdrawETH`) and the `depositETH` `onBehalfOf` (who receives aWETH) explicitly asserted since they are the recipient-in-bytes fields. Anchor the local `POOL_ABI`/`WETH_GATEWAY_ABI` selectors against the canonical Aave reference (compare the 4-byte selectors, or decode with the `@aave/*` published ABI already in the dependency closure) so a drifted local ABI fails rather than ships. Extend the existing approval assertion to decode and check the **spender**, not just the last-32-byte amount.

5. **Pin the V4 `sqrtPriceX96` price math against the Uniswap SDK (#318).** `uniswap/encoding.ts:381-392` computes price from `sqrtPriceX96 * sqrtPriceX96 / 2^192`. Add a test that derives the expected price for a representative pool via the Uniswap SDK's price utilities and asserts our `calculatePriceImpact` / price output matches within a tight band, so the hand-rolled fixed-point math has an external reference (the original intent of #318).

Throughout: every assertion must derive its expected value from the **reference** (protocol SDK or canonical ABI), never from the SDK's own quote or the same module/ABI that produced the bytes. That independence is the entire point of the ticket.

## Acceptance criteria / tests

- For a fixed swap intent (exact-in, exact-out, native-in), the Universal Router / V4 calldata produced by our `encodeUniversalRouterSwap` is byte-equal to the calldata produced by the Uniswap reference SDK for the same intent; the differential test fails if the two diverge. The reference SDK is a `devDependency` only and does not appear in the runtime closure.
- The V4 calldata's `TAKE_ALL` / `SETTLE_ALL` recipient is decoded and asserted; a swap intent with `recipient != msg.sender` either encodes that recipient (byte-equal to the reference) or **throws** a clear error — it must not silently pass with output routed to `msg.sender`. This turns F046/#444 into a failing test.
- The exact-in `amountOutMinimum` and exact-out `amountInMaximum` decoded from our calldata equal the reference SDK's computed bound (or the provider-derived bound), not a value recomputed by the same encoder under test.
- The Velodrome universal-router and CL encoder tests decode and assert the encoded recipient (matching the v2/leaf `args[3] === RECIPIENT` siblings); a non-sentinel caller recipient is asserted rejected/surfaced (F003/#444 becomes a failing test).
- The Morpho lend test no longer mocks `MetaMorphoAction`; it decodes the produced deposit/withdraw bytes with `blueAbi`/`erc4626Abi` and asserts `functionName`, `assets === amountWei`, `receiver === wallet`, `owner === wallet`, `to === vaultAddress`. Patching the path to use `withdraw(assets, ATTACKER, ATTACKER)` makes the test fail.
- The Aave lend tests decode `supply`/`withdraw`/`depositETH`/`withdrawETH` and assert every arg incl. native-ETH `to`/`onBehalfOf` and the approval **spender**. Putting the pool address in `onBehalfOf`, or the pool in `withdrawETH`'s `to`, makes a test fail.
- The local Aave `POOL_ABI`/`WETH_GATEWAY_ABI` and the Morpho selectors are anchored against the canonical protocol reference (selector or published-ABI comparison), so a drifted local ABI fails CI; reordering `supply`'s `(asset, amount, onBehalfOf, referralCode)` inputs makes the anchor test fail.
- The V4 `sqrtPriceX96` price math is asserted against a Uniswap-SDK-derived expected price within a tight band (#318).
- Full SDK `pnpm` typecheck / lint / test / build pass.

## Notes

- **Augments #318, does not open a new issue.** #318 is the swap-encoder min-out / differential-oracle cluster; this ticket is its test-infra realization plus the explicit `sqrtPriceX96`-vs-Uniswap-SDK pin called out in #318. The lend half (F189/F190/F160) extends the same differential-oracle principle to the lend signing path; record the augment color on #318 and flag it important for implementation.
- **Scope discipline.** This is differential testing and test hardening — it asserts the calldata matches the field values the SDK already knows it intended (recipient, min-out/max-in, amounts, `marketId`, `onBehalf`/`receiver`/`owner`, `deadline`). It does not guess user intent, does not broadly refuse to sign, and does not trust the RPC (the differential is offline: encode here, encode via the reference, compare bytes/fields). Permit2 signature payloads, where an encoder produces them, are in signing-path scope and follow the same decode-back-and-assert rule.
- **The Uniswap SDK is not currently installed.** Unlike `@aave/*` and `@morpho-org/*` (already in the dependency closure and reused on the borrow surface), no `@uniswap/*` package is in `pnpm-lock.yaml`. The differential oracle for swap therefore requires adding a new pinned `devDependency`; keeping it dev-only is mandatory so the deliberate hand-rolled-encoder supply-chain reduction (the reason there is no runtime Uniswap dependency) is preserved.
- **Relationship to the capstone.** This is one of the four `e2e-anvil-feature-test` blockers and supplies its interim, no-bundler independent-ABI decode oracles. This ticket anchors the **encode** step (offline, fast, no fork); the capstone then broadcasts real-signed transactions on an OP fork and asserts exact-amount on-chain recipient deltas. Landing this first means the e2e's oracles are already proven independent before the multi-day fork/bundler work begins.
- **Sibling consistency is the through-line.** The borrow-Morpho test already decodes with `blueAbi`; the Velodrome v2/leaf tests already assert the recipient. This ticket brings the swap-Uniswap, Velodrome-universal/CL, and both lend providers up to the same bar, removing the asymmetry that is itself why several of these bugs ship green.

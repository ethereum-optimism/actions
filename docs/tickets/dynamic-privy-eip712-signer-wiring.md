# Fix Dynamic/Privy raw-digest and EIP-712/Permit2 signer wiring

| Field | Value |
| --- | --- |
| **Severity** | medium |
| **Complexity** | 3 (of 5) |

| **Surface** | react Dynamic `createSigner` (hand-rolled raw-digest `sign()` with `0x` hex-stripping, connector backend vs `walletClient` backend, unchecked connector cast); react Privy `createSigner` (`signTypedData` cast to `CustomSource['signTypedData']` on the EIP-712/Permit2 seam) |
| **Resolves findings** | F030, F073, F062, F244 |
| **Candidate existing issue** | none |

## Problem

Two react hosted-wallet `createSigner` factories re-wrap a vendor signing account instead of returning it, and in doing so each introduces a divergence on a signing-path member that no test exercises. Both diverge from their node siblings, which return the vendor's viem account verbatim.

1. **Dynamic raw-digest `sign()` runs on a different backend with hand-rolled `0x` stripping.** The Dynamic factory builds a `LocalAccount` whose `signMessage`/`signTransaction`/`signTypedData` are bound to `walletClient`, but whose raw-digest `sign({ hash })` is hand-implemented against a different object: `connector.signRawMessage({ accountAddress, message: hash.startsWith('0x') ? hash.slice(2) : hash })`. It is the only provider that manually strips the `0x` prefix and routes raw-hash signing through the connector rather than the walletClient. viem calls `account.sign({ hash })` with a `0x`-prefixed 32-byte digest when this account is a Coinbase smart-wallet owner signing a UserOp hash. If `signRawMessage` expects the `0x` prefix or re-hashes/re-prefixes its input, the stripped bare-hex signs over the wrong digest, producing an owner signature that `validateUserOp` rejects on-chain or, worse, a valid signature over unintended data. The `connector` and the `walletClient.account` are two separate backend objects with no assertion they are bound to the same key, so a connector bound to account A while `walletClient.account` reports B yields a raw signature from A under the identity reported as B.

2. **Dynamic connector is force-cast with no capability guard.** `wallet.connector` is cast `as DynamicWaasEVMConnector` immediately after the `isEthereumWallet` check, which only confirms EVM capability, not that the connector exposes `signRawMessage`. A non-WaaS Ethereum Dynamic wallet (injected/embedded) passes the EVM guard, so the missing capability surfaces only when `sign()` is first invoked mid-flow rather than at construction.

3. **react Privy casts `signTypedData` across the EIP-712/Permit2 seam.** The Privy factory takes a fully-formed viem account from `toViemAccount(...)` and rebuilds a fresh `LocalAccount` via `toAccount(...)`, copying `address`/`sign`/`signMessage`/`signTransaction` uncast but casting `signTypedData` to `CustomSource['signTypedData']`. The cast is the only member requiring it, which is exactly the signal that Privy's `signTypedData` has a different param/generic shape than viem's `CustomSource` contract. A cast silences the compiler, not the runtime: viem invokes the casted member with the typed-data argument object it builds, and `signTypedData` is precisely the EIP-712 / Permit2 / 4337-typed-data signing surface. No test recovers a signer from a signature this member produces.

4. **None of the divergent members are covered.** Both react specs mock `viem/accounts.toAccount`, so the hand-rolled Dynamic `sign()` closure and the Privy `signTypedData` cast are never executed against a real digest or typed-data payload; the specs only assert the four `sign*` fields were passed through. There is zero `recoverAddress`/`verifyMessage`/`verifyTypedData` assertion anywhere under `packages/sdk/src/wallet`, so the exact bug the cast and the hand-rolled closure could hide ships green. The `validation.test.ts` locus that should encode signing-path intent is similarly thin (only `resolveSupportedChainIds` + `validateWalletAddress`).

Fund-safety framing: a Permit2 `PermitTransferFrom` signed through a mis-shaped `signTypedData`, or an ERC-4337 owner UserOp signed over a mis-stripped digest, either reverts at validation (recoverable availability loss, sponsored op burned) or, if the divergent backend produces a structurally valid signature over a payload other than the one the SDK believes it signed, authorizes a transfer/permit under the wrong digest. The SDK already holds the answer here: it can recover the signer from the signature it produced and assert it matches the reported address. This is fail-closed-where-the-SDK-already-knows plus sibling-consistency, not intent-guessing.

## Findings

- **F030** — Dynamic `createSigner` hand-rolls `sign({ hash })` via `connector.signRawMessage` with manual `0x` stripping (`hash.startsWith('0x') ? hash.slice(2) : hash`) on a different backend than its other three `sign*` methods, only reached on the smart-wallet owner path (`packages/sdk/src/wallet/react/wallets/hosted/dynamic/utils/createSigner.ts:28-37`).
- **F073** — react Privy `createSigner` re-wraps the vendor viem account and casts `signTypedData as CustomSource['signTypedData']` on the EIP-712/Permit2 seam with no recovering-signer test (`packages/sdk/src/wallet/react/wallets/hosted/privy/utils/createSigner.ts:22-29`).
- **F062** — Dynamic `wallet.connector` is cast to `DynamicWaasEVMConnector` with no `instanceof`/capability guard before `signRawMessage`; `isEthereumWallet` only confirms EVM capability, so a non-WaaS connector throws only at first `sign()` call (`packages/sdk/src/wallet/react/wallets/hosted/dynamic/utils/createSigner.ts:24-25`). The cross-flow elaboration confirms `sign()` is invisible in EOA flows and only exercised when the account is a 4337 owner.
- **F244** — `validation.test.ts` (and the two react signer specs) is the missing-coverage locus: it must grow per-member intent-encoding blocks (Rule 9); today the signer specs mock `toAccount` so neither divergent member runs, and `validation.test.ts` covers only two validators (`packages/sdk/src/utils/__tests__/validation.test.ts:1-51`; signer specs at `.../dynamic/utils/__tests__/createSigner.spec.ts` and `.../privy/utils/__tests__/createSigner.spec.ts`).

## Root cause

Both react factories re-wrap the vendor account rather than returning it. The node siblings do not: node Privy returns `createViemAccount(...)` directly and node Turnkey returns `createAccount(...)` directly, so their `signTypedData` and raw-digest `sign` are the vendor's correct-by-construction implementations and need no cast. The react Privy factory re-asserts the account through `toAccount(...)`, and the only member whose shape does not survive the rewrap is `signTypedData`, papered over with a cast. The react Dynamic factory goes further: it sources `address` from `walletClient.account.address` but routes raw-digest `sign` to a second backend (`connector.signRawMessage`), reconstructs the message encoding by hand, and casts the connector unchecked. Two mechanically independent backends, a hand-rolled digest transform, and a compile-time cast all sit on signing-critical members with no recovery test to prove any of them produce a signature that recovers to the reported address.

## Recommended approach

SDK signing-path work: align the two react closures with their node siblings and with each other, and add recover-from-signature coverage so the divergent members can no longer ship unverified. Permit2 signature payloads are in signing-path scope, so the `signTypedData` seam is in scope here.

1. **react Privy: drop the rewrap, or verify the cast (F073).** Prefer returning the genuine Privy viem account directly (`return privyViemAccount`) when it already satisfies `LocalAccount`, which removes the cast and preserves Privy's correct-by-construction `signTypedData`. If the rewrap is genuinely required (e.g. to drop capabilities), keep it but add a recovery test that signs a representative Permit2 `PermitTransferFrom` struct through the rewrapped member and asserts `verifyTypedData({ address: signer.address, ... })`. Normalize the copied `address` via `getAddress` so the rewrap cannot silently widen the address shape.

2. **Dynamic: guard the connector and reconcile the backend before relying on `sign()` (F062, F030).** Add a narrow capability check (`typeof connector?.signRawMessage === 'function'`) before the cast and throw a descriptive error at construction if absent, instead of an unchecked cast that fails opaquely mid-flow. Verify against the Dynamic WaaS connector contract whether `signRawMessage` expects a `0x`-prefixed hash and whether it re-hashes/re-prefixes; do not hand-strip `0x` unless the contract requires the bare form. Where the Dynamic SDK exposes its own viem account abstraction, prefer wiring all `sign*` members (including raw `sign`) to a single backend so `signTypedData` (walletClient) and `sign` (connector) cannot be backed by two different keys; at minimum assert the connector signs for `walletClient.account.address`.

3. **Cover the divergent members (F244).** Both react specs currently mock `viem/accounts.toAccount`, so add tests that exercise the real closures: for Dynamic, assert the exact `message` bytes handed to `connector.signRawMessage` match what viem hands `account.sign({ hash })` (catches the `0x`-strip divergence) and that a recovered signer matches the reported address; for Privy, sign a typed-data payload through the rewrapped `signTypedData` and assert it recovers. Grow `validation.test.ts` toward per-validator intent-encoding blocks as the structural twin of these signer-coverage additions, so a future widening of any signing-path guard fails CI. The full real-credential recovery (real Privy/Turnkey/Dynamic creds + Anvil-simulated signing, including the Permit2 `verifyTypedData` and the `0x`-prefixed-vs-bare digest cases) lands in the single consolidated Anvil feature-test ticket, not here; these interim unit tests anchor the closures against a mock-backed signer.

This is sibling-consistency and fail-closed-where-the-SDK-already-knows, not refuse-to-sign or RPC-trust hardening. The connector-vs-walletClient address reconciliation in step 2 depends on the shared hosted-signer-address-reconcile seam (the abstract `HostedWalletProvider` self-test); see Blocked by.

## Acceptance criteria / tests

- A react Privy signer either returns the vendor viem account directly (no `signTypedData` cast remains) or, if rewrapped, a test signs a representative Permit2 `PermitTransferFrom` typed-data struct through `signer.signTypedData` and asserts `verifyTypedData({ address: signer.address, domain, types, primaryType, message, signature })` is true. A test that mutates `signTypedData` to return a signature for a different struct must now fail.
- The Dynamic factory throws a descriptive, construction-time error when `wallet.connector` lacks `signRawMessage` (capability guard), rather than throwing opaquely at first `sign()` call. A test passes a connector without `signRawMessage` and asserts the typed construction error.
- A Dynamic signer test (with `toAccount` no longer mocked, or with a real closure invoked) asserts the exact `message` argument passed to `connector.signRawMessage` matches the digest viem hands `account.sign({ hash })`, encoding whether `0x` is stripped per the connector contract; a recovered signer from that raw signature matches `walletClient.account.address`. A test that flips the `0x`-strip behavior must fail.
- `validation.test.ts` gains per-validator intent-encoding `describe` blocks for the signing-path validators (range/finiteness/canonical-zero/expiry/same-asset), each asserting its invariant so a future widening fails CI.
- The full real-credential recovery assertions (each provider's `signMessage`/`signTypedData` recovers to `wallet.address`, the Permit2 witness recovery, and the Dynamic `0x`-prefixed-vs-bare digest case) are deferred to the single consolidated Anvil feature-test ticket; this ticket does not file additional e2e tickets.

## Notes

- The `sign()` closure on the Dynamic account is unreachable in EOA flows: `EOAWallet.send` signs via `signTransaction` (walletClient), so the hand-rolled raw-digest path is only exercised when this `LocalAccount` is a Coinbase smart-wallet owner and viem calls `account.sign({ hash })` on a UserOp hash. That is why the divergence is invisible today and why interim coverage must drive the smart-wallet owner path, not just an EOA send.
- The Dynamic SDK (`@dynamic-labs/waas-evm`) is a peer dependency not installed in this repo, so the exact `signRawMessage` encoding contract (prefix expectation, re-hashing) must be confirmed against Dynamic's connector docs before deciding whether the `0x` strip is correct or a bug. Until confirmed, treat the strip as suspect and assert the produced bytes in a test.
- The connector-vs-`walletClient` key reconciliation (assert the connector signs for `walletClient.account.address`) is the same identity-reconciliation seam the abstract `HostedWalletProvider` self-test owns (F074). That shared seam is the `hosted-signer-address-reconcile` blocker: land the per-provider sign-closure alignment here, then have the reconciliation self-test cover all hosted providers uniformly so this fix does not become a one-off.
- node Privy and node Turnkey are the consistency reference: both return the vendor's viem account verbatim with no rewrap and no cast. The cleanest fix collapses the react factories toward that same shape; the rewrap should survive only if it provides something the direct return cannot, and if it does, the casted member must carry a recovery test.

## Problem

Every hosted wallet (Privy, Turnkey, Dynamic) reports a `.address` and signs with a backend key, and the SDK never proves the two agree. The reported address flows into `from`, balance lookups, ERC-20 approval owners, lend `onBehalfOf`, swap/transfer recipients, and the smart-wallet owner-index resolution that decides which on-chain owner slot signs a 4337 UserOp. A reported address that does not correspond to the signing key is a fund-relevant divergence: approvals and recipients target one account while the key signs as another, surfacing either as a wrong-slot `ownerIndex` (reverting UserOp) or as funds moved relative to the wrong account, with no construction-time error.

The companion production ticket (`hosted-signer-address-reconcile`) adds the reconciliation seam. **This ticket is the test sibling**: the reason the divergence persists today is that *every* hosted-wallet unit test mocks the vendor signing primitive and asserts `signer.address === <the address fed into the mock>`, so the assertions are tautologies that cannot fail when the address-handling contract regresses. A repo-wide `grep -rE 'recoverAddress|recoverMessageAddress|verifyMessage|verifyTypedData' packages/sdk/src/wallet` returns **zero matches** — no hosted-wallet test ever recovers a signer from a real signature. With no conformance test, the reconciliation seam (once built) has nothing pinning its behavior, and a future refactor can silently re-introduce address-from-caller trust.

These are unit/conformance tests using the **in-hand test-mode Privy/Turnkey/Dynamic credentials** with locally known keys, so a probe signature can be recovered and compared to the reported address per provider × runtime. They are deliberately scoped *below* the consolidated real-credential Anvil e2e (`e2e-anvil-feature-test`), which depends on this ticket landing first.

## Findings

- **F028** — `packages/sdk/src/wallet/node/providers/hosted/privy/PrivyHostedWalletProvider.ts:87-95`: node-Privy `createSigner` spreads `{ ...params }` (caller `address` reaches the signer unreconciled) while sibling `toActionsWallet` (`:67`) runs `getAddress`; the two methods have divergent address contracts and neither reconciles against `walletId`.
- **F029** — `packages/sdk/src/wallet/node/wallets/hosted/privy/PrivyWallet.ts:36`: `this.address = params.address` (caller input), then `performInitialization` (`:49-50`) builds the signer from `walletId`; reported address and signing key are never reconciled (the React Privy sibling derives address from the signer, so the two harnesses must diverge).
- **F030** — `packages/sdk/src/wallet/react/wallets/hosted/dynamic/utils/createSigner.ts:28-37`: the hand-rolled `sign({ hash })` closure strips an optional `0x` prefix and routes to `connector.signRawMessage` — a different backend than `signMessage`/`signTransaction`/`signTypedData` (`walletClient.*`); this is the EIP-712 / Permit2 digest path and is captured-but-never-invoked by the spec (`createSigner.spec.ts:52,60-67`, F225).
- **F031** — `packages/sdk/src/wallet/node/wallets/hosted/turnkey/utils/createSigner.ts:25-31`: `ethereumAddress` and `signWith` are forwarded to `createAccount` with no format validation and no cross-check that the resolved `signWith` key controls `ethereumAddress`; the only test (`createSigner.spec.ts:53-76`, F227) asserts the value was forwarded to the mock, confirming the bypass rather than its safety.
- **F074** — `packages/sdk/src/wallet/core/providers/hosted/abstract/HostedWalletProvider.ts:45-56`: no shared signer-identity reconciliation seam across providers; the F231 refinement (`packages/sdk/src/wallet/core/wallets/eoa/EOAWallet.ts:44-52`) notes no hosted wallet has a construction-time signer-address self-test, so reported `.address` vs actual key is never verified for any provider.
- **F090** — `packages/sdk/src/wallet/core/wallets/smart/default/utils/findSignerInArray.ts:17-29`: `getAddress()` is called inside `findIndex`, so a malformed `signers[]` entry throws `InvalidAddressError` instead of returning `-1`; the suite (`findSignerInArray.spec.ts:9-68`, F239) covers only well-formed arrays, so the resolved-signer → owner-index binding has no malformed-input test.
- **F107** — `packages/sdk/src/wallet/core/wallets/smart/default/utils/findSignerIndexOnChain.ts:39-65`: the index scan is driven by monotonic `nextOwnerIndex`, but the load-bearing invariant is only enforced by a mislabeled test (`findSignerIndexOnChain.spec.ts:20-22,45-48` comments say `ownerCount`, F235); every fixture sets `ownerCount === nextOwnerIndex`, so an `ownerCount`-based regression that skips a live high-slot owner stays green.

## Root cause

Two layers are untested in a way that hides the binding bug. First, at the signer layer, every hosted test asserts the mock against itself: `createMockPrivyClient` is an empty stub that cannot resolve `walletId → key`, so `signer.address === hostedWallet.address` is an identity-in-equals-identity-out tautology (F226/F232); Turnkey's `ethereumAddress` is only asserted as forwarded-to-mock (F227); Dynamic's raw-digest `sign` closure is never invoked because `toAccount` is mocked (F225). Because the vendor primitive is mocked, no test ever derives an address from the signing key and compares it to the reported `.address`. Second, at the owner-index layer, the resolved signer feeds `findSignerInArray` / `findSignerIndexOnChain`, but those suites cover only well-formed, gap-free owner sets, so the address the signer reports is never exercised against the slot-resolution logic that consumes it. The fix space is in-scope: an obvious conformance check (recover-equals-reported-address; resolved-signer maps to the correct owner slot) that the SDK already has the inputs to make, missing in every provider and inconsistent across the node/react siblings.

## Recommended approach

Add provider × runtime conformance tests that bind the *reported* address to the *signing* key with a real recovery, and that bind the *resolved signer* to the correct owner-index slot — using the in-hand test-mode credentials with locally known keys so signatures are recoverable. No production behavior is asserted beyond what the `hosted-signer-address-reconcile` seam introduces; this ticket pins that seam and the owner-index consumption so they cannot silently regress.

1. **Signer-identity conformance (per provider × runtime).** For each of node-Privy, react-Privy, Turnkey (node + react), and Dynamic (react), construct the signer from a test-mode credential backed by a **known local key**, sign a fixed domain-separated probe message, and assert `recoverMessageAddress(signature) === getAddress(signer.address) === getAddress(wallet.address)`. This replaces the empty-mock tautology (F226/F232) with a real recovery. The node-Privy `createSigner` vs `toActionsWallet` asymmetry (F028) is covered by running the same assertion through *both* public entry points so they cannot diverge.
2. **Adversarial wrong-address (node-Privy F029, Turnkey F031).** Construct node-Privy with a correct `walletId` but a deliberately wrong `address`, and Turnkey with a correct `signWith` but a wrong `ethereumAddress`; assert the construction rejects (once the seam lands) or, at minimum, that `recoverMessageAddress(probe) !== wrong-address` so the divergence is observable. Add a malformed-`ethereumAddress` / malformed-`address` case asserting it fails the `isAddress`/`getAddress` shape check loudly rather than flowing through.
3. **Dynamic raw-digest closure recovery (F030).** Do **not** mock `toAccount`. Stub `connector.signRawMessage` with a deterministic local key, call `signer.sign({ hash })`, and assert `recoverAddress({ hash, signature }) === walletClient.account.address`. Add both the `0x`-prefixed and bare-digest cases so the hex-stripping branch and the cross-backend `signRawMessage`-vs-`walletClient` divergence have behavioral coverage. Permit2 / EIP-712 typed-data payloads are in signing-path scope: sign a real `PermitTransferFrom` struct through the provider's `signTypedData` and assert `verifyTypedData({ address: signer.address, … })`.
4. **Resolved-signer → owner-index binding (F090, F107).** Assert the signer the hosted path produces maps to the correct owner slot. For `findSignerInArray` (F090, F239): add a malformed/non-address `signers[]` entry and pin the contract (throws today; `-1` is the desired fail-closed shape) plus a WebAuthn-only set (returns `-1`, construction fails). For `findSignerIndexOnChain` (F107, F235): add the load-bearing fixture where an owner removed from a low slot leaves a live owner in a high slot (`ownerCount < nextOwnerIndex`), so an `ownerCount`-based regression that skips the live high-slot owner fails; fix the mislabeled `// ownerCount` comments to `nextOwnerIndex`.
5. **`isLocalAccount` collision (links F074).** A hosted-derived signer is itself `type: 'local'`; add a test that passing it into `WalletNamespace.toActionsWallet` does not silently route to a bare `LocalWallet` that skips reconciliation (the seam-bypass hole, F215). This pins the routing branch the reconcile-seam ticket closes.

Tests encode **intent, not behavior**: each must be able to fail when the address-handling contract regresses (recover-equals-address, signer-maps-to-correct-slot), which the current mock-against-itself assertions cannot. Gate the credential-backed cases on the test-mode creds being present (skip-with-warn locally, fail-fast in CI) so a missing credential is loud, not a silent green.

## Acceptance criteria / tests

- For **each provider × runtime** (node-Privy, react-Privy, node-Turnkey, react-Turnkey, react-Dynamic), a conformance test signs a probe with a known test-mode key and asserts `recoverMessageAddress(sig) === getAddress(signer.address) === getAddress(wallet.address)`. No assertion compares the signer to the mock that produced it.
- Node-Privy `createSigner` and `toActionsWallet` are both covered by the recovery assertion, so the F028 asymmetry cannot reappear; the empty-`MockPrivyClient` tautology (`:43-44`, `:120-141`) is gone.
- Node-Privy with correct `walletId` + wrong `address`, and Turnkey with correct `signWith` + wrong `ethereumAddress`, are rejected (or never report the wrong address); malformed address inputs fail an `isAddress`/`getAddress` shape check loudly.
- Dynamic: the raw-digest `sign({ hash })` closure is invoked (not mocked) and `recoverAddress({ hash, sig }) === walletClient.account.address` for both `0x`-prefixed and bare digests; a real Permit2 `PermitTransferFrom` signed via `signTypedData` passes `verifyTypedData`.
- `findSignerInArray`: a malformed/non-address owner entry has a pinned contract (throw vs `-1`) and a WebAuthn-only set is covered (F090); `findSignerIndexOnChain`: an `ownerCount < nextOwnerIndex` removal-gap fixture fails an `ownerCount`-based regression, and the test comments label `nextOwnerIndex` correctly (F107).
- Passing a hosted-derived (`type: 'local'`) signer into `WalletNamespace.toActionsWallet` does not silently bypass reconciliation (links F074/F215).
- Each new test can fail when the address-handling / owner-index contract regresses (intent, not tautology); credential-backed cases fail fast in CI when test-mode creds are absent rather than skipping silently.

## Notes

- **Severity medium.** The divergence is real and fund-relevant, but the dominant trigger is integrator misconfiguration (an inconsistent `(walletId, address)` or `ethereumAddress`/`signWith` pair), not an external-attacker primitive. The conformance tests convert a silent wrong-account signing into a loud, observable failure.
- This is the **test sibling** of `hosted-signer-address-reconcile` (the wave-0 production seam fix). That ticket adds the `reconcileSignerAddress` seam; this ticket pins it with real recovery and keeps the owner-index consumption honest. The two are intentionally split so the seam has conformance coverage from day one.
- **This ticket blocks `e2e-anvil-feature-test`**, the single consolidated Anvil capstone. The capstone's wallet-hosted slice (F230) drives the *real-credential* recovery against an OP fork; the unit/conformance seam here is its prerequisite and stands on its own without the fork harness.
- The differential / hardening framing here is security review and conformance testing, not an audit; the goal is to make the address-binding contract falsifiable.
- F068-priv (the react-Privy vendor re-wrap `signTypedData` CustomSource cast) and the broader EIP-712/Permit2 signer wiring are owned by `dynamic-privy-eip712-signer-wiring`; this ticket only requires the message-recover and Permit2 `verifyTypedData` conformance assertions, not the wiring refactor.
